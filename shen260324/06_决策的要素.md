# 第6章：决策的要素
> 沈老师视角 · 2026-03-24

这章讲决策的结构。关键洞见：决策不是「选择」，是「状态机」——有五个步骤，缺一步都不算完成决策。另一个洞见：推行是决策的一部分，不是决策之后的事。

---

## 一、本章核心流图（决策状态机）

```mermaid
flowchart TD
    TRIGGER["触发：需要做一个决策"]

    TRIGGER --> S1["第一步：判断问题性质\n这是第几次出现类似情况？"]

    S1 -- "多次出现或预期还会出现" --> RECUR["经常性问题\n正确响应：建立通则/规则\n下次遇到直接查表，不重新决策"]

    S1 -- "真正首次，不可能再现" --> EXCEPT["真正的例外\n正确响应：一次性方案"]

    RECUR & EXCEPT --> S2["第二步：定义边界条件\n这个决策必须满足什么？\n不能损害什么？\n写成可检验的格式"]

    S2 --> S3["第三步：确定「正确」的方案\n不是「可接受」的方案\n不是「大家都满意」的方案\n而是「在边界条件内最有效」的方案"]

    S3 --> S4["第四步：设计推行方案\n推行是决策的一部分，不是后续步骤\n必须回答三个问题：\n① 谁执行？\n② 执行者有能力和资源吗？\n③ 反馈验证点在哪里（预设时间节点）？"]

    S4 --> S5{推行方案可行？}

    S5 -- "回答不了① ② ③" --> INCOMPLETE["决策未完成\n回到 S3 重新考虑\n这不是一个可推行的决策"]

    S5 -- "可行" --> EXECUTE["执行，并等待反馈信号"]

    EXECUTE --> FEEDBACK["在预设节点检验：\n结果是否符合预期？\n边界条件是否还成立？"]

    FEEDBACK -- "反馈节点触发修正" --> S3
    FEEDBACK -- "按预期进行" --> DONE["决策完成"]

    style RECUR fill:#f0a500,color:#fff
    style INCOMPLETE fill:#e74c3c,color:#fff
    style DONE fill:#4ecdc4,color:#fff
```

## 二、关键概念裁判

### 经常性 vs 例外：最重要的第一步

**典型错误**：线上出现了数据不一致 bug，紧急处理：定位问题，修数据，发公告，关单。这是一次有效的决策处理吗？

第一反应：是，响应及时，问题解决了。

**哪里错了**：

```mermaid
flowchart TD
    BUG1["第一次数据不一致\n紧急处理，修数据，关单"]
    BUG2["第二次同类型问题\n又是紧急处理，修数据，关单"]
    BUG3["第三次同类型问题\n再一次紧急处理..."]

    BUG1 --> BUG2 --> BUG3

    BUG3 --> QUESTION["第三次出现时才意识到：\n根因是什么？\n是数据模型设计？\n服务间缺少事务保证？\n补偿逻辑有漏洞？"]

    BUG1 & BUG2 & BUG3 --> COST["三次的总代价：\n3次紧急处理成本\n3次业务方投诉\n3次不一定一致的修复方案\n系统可靠性信任损失"]

    QUESTION --> RULE["如果第一次就识别为经常性问题：\n制定分布式事务使用规范\n建立自动检测机制\n→ 第二次、第三次不会发生"]

    style COST fill:#e74c3c,color:#fff
    style RULE fill:#4ecdc4,color:#fff
```

**复杂度分析**（工程视角）：
- 对着信号动作（每次当例外处理）= O(n) 代价
- 建立通则（根因修复 + 规范建立）= O(1) 查表代价

用 O(n) 做可以 O(1) 的事是性能问题，也是决策有效性问题。

**经常性问题的识别信号**：这类问题出现超过一次，或者你预期它还会出现。一旦识别为经常性，应该建立通则，不是应急处理。

---

### 边界条件：决策的约束定义

这是最难操作化的概念，但也是最关键的一步。

```mermaid
flowchart LR
    WHAT["边界条件是什么？\n决策必须满足的约束\n不能损害的目标\n在任何情况下都成立的前提"]

    WHAT --> FORMAT["写法要求：\n不能写成模糊陈述\n必须写成可检验的格式\n= 给定任何方案\n能判断它是否满足这个条件"]

    FORMAT --> GOOD["好的边界条件写法：\n① 这个方案不能增加服务间的耦合度\n② 这个决策三年后必须还能被更改\n③ 不能以损害用户数据安全为代价"]

    FORMAT --> BAD["差的边界条件写法：\n① 需要兼顾各方利益\n② 要考虑长期影响\n③ 要综合考量"]

    BAD --> PROB["差的边界条件的问题：\n无法检验是否满足\n任何方案都可以声称满足\n= 没有约束力，只是愿望"]

    style GOOD fill:#4ecdc4,color:#fff
    style BAD fill:#888,color:#fff
    style PROB fill:#e74c3c,color:#fff
```

**一个诊断性问题**：给定你当前最重要的决策，把它的边界条件写出来。能不能对任意一个方案，回答"它满足 / 不满足这个条件"？如果不能，边界条件还没写清楚。

### 推行是决策的一部分

```mermaid
flowchart TD
    TRAP["常见陷阱：\n做完决策 → 宣布决策 → 等别人推行\n把「决策」和「推行」当成两件事"]

    DRUCKER["德鲁克的洞见：\n一项决策如果不能付诸行动\n就不是决策，是良好意愿\n推行是决策本身的一部分"]

    TRAP --> DRUCKER

    DRUCKER --> CHECK["推行检查清单：\n① 谁执行？（必须有具体的人，不能是「团队」）\n② 执行者有能力和资源吗？（没有 = 决策还没完成）\n③ 反馈验证点在哪里？（必须预设，不能等出问题了再看）"]

    CHECK --> EXAMPLE["例：一个系统重构决策\n不完整的版本：「我们决定要重构这个服务」\n完整的版本：\n  执行者：张三（有该系统的核心理解）\n  资源：下个 sprint 的 60% 工时\n  反馈节点：30天后评估进度，60天后验收边界条件①②"]

    style TRAP fill:#888,color:#fff
    style DRUCKER fill:#ff6b6b,color:#fff
```

## 三、同构识别

**状态机（State Machine）↔ 决策的五个步骤**

决策的五个步骤是一个状态机：问题性质识别 → 边界条件定义 → 方案确定 → 推行设计 → 反馈验证。非法的状态跳转（跳过某步直接进入下一步）导致决策失效。

最常见的非法跳转：
- 跳过边界条件（S2）直接做方案（S3）→ 方案满足了什么约束？不知道
- 跳过推行设计（S4）直接宣布执行（S5）→ 谁做、怎么做不清楚，决策停在纸上

**O(n) → O(1) 复杂度优化 ↔ 建立通则**

每次遇到经常性问题都重新分析和决策 = O(n)，n 是问题出现次数。建立通则，问题出现时直接查表 = O(1)。通则建立的一次性成本 = O(k)，k 是通则的复杂度，通常 k << n × 单次分析成本。

## 四、可执行模型

```
IF 遇到一个问题需要决策
THEN 第一个问题：这类问题之前出现过吗？以后会再出现吗？
     YES → 经常性问题，目标是建立通则，不是解决这一次
     NO  → 真正的例外，一次性方案

IF 准备做一个重要决策
THEN 先写边界条件：
     格式：「这个方案不能 X / 必须满足 Y」
     测试：给定任意方案，你能判断它满足还是不满足吗？
     不能判断 = 边界条件还没写清楚

IF 一个决策已经确定了方案
THEN 在宣布之前，先回答三个问题：
     ① 谁执行？（具体的人）
     ② 执行者有能力和资源吗？
     ③ 什么时候检验结果？（具体的时间节点）
     回答不了 → 决策还没完成

IF 反复遇到同类型的问题（>2次）
THEN 识别为经常性问题
     停止应急处理，切换到「建立通则」模式
     根因分析 → 制定规范 → 下次直接查表
```
