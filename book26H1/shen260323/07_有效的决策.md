## 第二步：裁判循环

### 从意见开始

**第一反应（错的）**：从意见开始不就是确认偏见吗？

我读到这里的第一直觉是拒绝——"从意见开始，然后用事实检验"，这不就是先入为主？先有结论再找证据？这跟我教学生的科学思维完全相反。

**哪里错了**：

确认偏见的问题是：你不知道你有假设，所以你不知道应该检验什么。你的假设是隐性的，所以事实的选择、数据的解读都在无意识地向你的隐性假设靠拢，而你以为自己在"客观地看数据"。

"从意见开始"恰恰相反：把假设显式化。

这两种方法的关键区别：
- 确认偏见：假设是隐性的 → 你不会问"什么证据能证明我是错的"
- 德鲁克方法：假设是显式的 → 你必须问"什么证据能证明这个假设是错的"，然后去找那个证据

**TDD的对应**：

这就是TDD（测试驱动开发）的逻辑。

TDD要求你先写测试（形成假设：这段代码应该有这样的行为），然后写代码让测试通过（收集证据）。如果测试失败，你更新代码，不是更新测试（不能为了让测试通过而修改测试——这就是确认偏见）。

"从意见开始"= 先写测试
"用事实检验" = 运行测试，看是否通过
更新意见 = 重构代码让测试通过（不是修改测试来迎合当前代码）

TDD的核心价值不是"先写测试"，而是"先定义预期行为，再检验实现"。德鲁克在说同样的事：先定义你认为世界是什么样的，然后让现实来检验，而不是看了一堆数据之后再说自己"发现"了什么规律（那个规律可能是你的隐性假设投影出来的）。

**教学例子**：

我在批改作业时，有时候看到一个学生的代码，第一眼觉得"这个思路有问题"。

差的方式：带着这个感觉往下读，会越来越觉得有问题（确认偏见）。
好的方式：把这个感觉显式化——"我认为这段代码在X情况下会出错"——然后构造那个X的测试用例，看是否真的出错。有时候代码是对的，只是写法不熟悉；有时候确实出错了，而且在我没想到的地方。

显式的假设让裁判变得精确，也让我能够被事实纠正。

### 反面意见机制

**为什么需要制度，不能靠个人自觉？**

一个正确的少数派意见，在组织内部会自然消亡。原因：
1. 权力结构压制异见（下级不敢公开反驳上级）
2. 确认偏见是默认设置（人会自动过滤不符合期望的信息）
3. 社会成本（唱反调的人承担社会代价）

反面意见机制的最小可行版本：

指定一个人（可以轮换）在决策前做adversarial testing：
1. 整理所有支持这个决策的理由
2. 构造最强力的反驳
3. 决策者必须书面回应这个反驳（不是推翻，是说明"为什么在存在这些反驳的情况下仍然选择"）

**教学应用**：

在设计新课程或重大课程改革前，我应该让另一个老师扮演"魔鬼代言人"——不是随机提意见，而是系统性地构造"为什么这个改革会失败"的论证，然后我必须回应每一条。

我通常不做这个。我设计课程，自己觉得合理，就推进了。这是结构性的风险——没有压力测试就上线。

### "什么都不做"是决策

**这是我最没有用过的选项。**

两个判断标准：
1. 行动的利益 > 成本+风险？
2. 什么都不做的代价，和行动的风险相比，哪个更大？

如果两个问题的答案都指向"不行动"，那什么都不做就是正确决策。不是拖延，是主动选择。

**教学例子**：

有一年我考虑给课程加一个大型项目作为期末评估（代替考试）。理由：项目能更真实评估综合能力。

用两个标准检验：
- 利益大于成本+风险吗？项目评估确实更全面，但批改成本从2天变成2周，且项目作弊更难发现。
- 什么都不做（保持考试）的代价？考试的局限性是已知的，但是已经被管理了（题型设计）。

什么都不做，是正确决策。不是因为懒，是因为分析后成本大于收益。

## 第三步：结构

```
1. 形成意见（显式假说）
    ↓
2. 问：什么证据能证伪这个假说？
    ↓
3. 内置反面意见机制（adversarial testing）
    ↓
4. 用事实检验（不是用事实验证）
    ↓
5. 被证伪 → 修正意见，重新循环
   未被证伪 → 进入判断
    ↓
6. 判断：行动的利益 > 成本+风险？ + 什么都不做是否更好？
    ↓
7. 行动 → 进入第6章的推行流程
   不行动 → 设定重新评估的触发条件
```

---

## 第四步：可执行模型

```
IF 需要做重大决策
THEN 先写下意见（假说），再问：什么证据能证明我是错的？去找那个证据

IF 只听到赞成声音
THEN 主动指定一个人做adversarial testing，要求书面反驳

IF 形成了决策方案
THEN 先问：什么都不做，后果是什么？如果后果可以接受，不做

IF 意见通过了检验，准备行动
THEN 进入第6章推行流程：谁执行？能力够？验证点在哪里？
```

## 第五步：接入CS知识体系

**同构：TDD（测试驱动开发）**

| TDD | 德鲁克决策 |
|-----|-----------|
| 先写测试（定义预期行为） | 先形成意见（定义假说） |
| 运行测试，观察失败 | 寻找证伪证据 |
| 写代码让测试通过 | 收集数据，修正意见 |
| 不修改测试来迎合代码 | 不选择性忽视反驳意见 |
| 重构（优化实现，保持测试） | 推行决策（改进执行，保持方向） |

确认偏见 = 为了让测试通过而修改测试，而不是修改代码。

**同构：adversarial testing / red teaming**

软件安全里有一个实践：红队（red team）专门尝试攻破系统，找漏洞。不是随机测试，是系统性地站在攻击者的视角构造攻击。

反面意见机制 = 组织决策层面的red teaming。

一个没有经过red team测试的软件系统，上线后被攻击，这是可预见的失败。一个没有反面意见机制的组织决策，同样是未经压力测试就上线。

**接入教学**：

"从意见开始"对学生的最大应用：

我现在教学生分析问题时，要求他们先写下"我认为这个问题的解法是X，理由是Y"，再开始搜索验证。不允许"先查资料再形成观点"的顺序。

为什么？因为先查资料再形成观点，你学到的是别人的结论，没有经历你自己的假说→检验→修正过程。那个过程才是真正的理解形成，不是结论的接收。

这是"理解 = 行为能力，不是语言能力"在教学设计上的直接应用。
