# 第3章：我能贡献什么

## 第零步：ER图

```
管理者 {
    string 思维起点  "我能贡献什么（而非我能得到什么）"
}

贡献 {
    enum 类型       "直接成果 | 价值标准建立 | 培养后继者"
    string 接收方   "组织外部 or 组织整体"
}

管理者 ||--o{ 贡献 : "定义并实现"
贡献_直接成果    }|--|| 贡献 : "子类"
贡献_标准建立    }|--|| 贡献 : "子类"
贡献_培养后继者  }|--|| 贡献 : "子类"
```

简单。贡献是三个子类的父类，管理者的工作方向是"贡献"，而不是"任务完成"。

---

## 第一步：概念自评

| 概念 | 评分 | 备注 |
|------|------|------|
| 贡献导向（思维转换） | 1 | 能说，但压力下会退回"完成任务"模式 |
| 三类贡献 | 2 | 直接成果清楚，标准建立和人才培养操作上还模糊 |
| 有效会议 | 2 | 知道条件，执行上还不稳定 |

---

## 第二步：裁判循环

### 贡献导向

**关键洞察**：把自己的工作想成实现一个接口（interface），而不是执行一个实现（implementation）。

问题不是"我能构建什么"，而是"系统需要我实现什么接口"。

这个切换很重要。"我能构建什么"是以自己的能力为出发点，"系统需要什么接口"是以外部需求为出发点。

**教学例子（第一直觉，错的）**：

我最初设计数据结构课程时，是从"我擅长讲什么"出发的：链表、树、图，讲得非常系统，逻辑自洽。这是高效率的课程交付。

但问题是：学生学完这门课，能做什么？他们能选择合适的数据结构解决实际问题吗？

一次考试之后，我发现大部分学生能背出各种结构的时间复杂度，但给一个实际问题，他们不知道该用哪个。

我的课程实现了我定义的规格说明，但这个规格说明本身是错的——我在设计我想教的，不是学生需要能做到的。

重新定义接口之后：这门课结束时，学生面对一个新问题，需要能独立选择合适的数据结构并说明理由。这个新接口反过来改变了我讲什么、怎么讲、考什么。

**正例**：
- 写代码前先写函数签名和docstring，强迫自己先定义接口
- 开始一个项目前先问：这个项目结束时，用户能做到什么？

**反例伪装**：
- "我完成了所有交代给我的任务" → 完成任务是实现了别人定义的接口，贡献是主动定义接口

---

### 价值标准建立（最被低估的一类）

**关键结构**：
```
管理者的每一个具体决定 → 向下传递一个信号 → 信号累积 = 实际标准
（说的标准）- （做的标准）= 标准债务，组织每天在还息
```

**教学例子**：

我在课上说"代码可读性和正确性同等重要"。但我批改作业时，如果代码逻辑对但变量名一团糟，我仍然给满分。

我刚才做的是什么？我更新了学生对"这门课实际标准是什么"的认知。不是我说的那个标准，是我做的那个标准。

这不是师德问题，这是信息传递的基本原理。行为比语言更直接，因为行为不能被误解。

---

## 第三步：结构

```
interface 管理者的贡献 {
    直接成果(): 短期可见的组织输出
    建立标准(): 通过日常决定传递的行为规范
    培养后继者(): 下一代知识工作者的能力建设
}
```

三类贡献的时间尺度完全不同：直接成果是季度级，标准建立是年级，人才培养是五年级。如果只追踪直接成果，后两类会被系统性忽视。

---

## 第四步：可执行模型

```
IF 开始任何工作前
THEN 先写：这件事完成后，外部世界（用户/学生/组织）会有什么不同？说不出来则重新定义工作

IF 作为老师做了一个与自己宣称标准不符的决定（比如对低质量作业网开一面）
THEN 意识到：你刚刚更新了学生对实际标准的理解，不是你说的那个

IF 要开一个会
THEN 先写：这次会议结束时，[什么结果]将会[存在/改变]。写不出来则不开
```

---

## 第五步：接入CS知识体系

**同构：interface vs implementation**

面向对象设计的核心原则之一：针对接口编程，不针对实现编程。

贡献导向就是把"我的工作"想成一个interface的实现：
- 接口由系统需求定义（外部贡献方向）
- 实现由我的能力决定（内部执行）
- 评估标准在接口层，不在实现层

如果你只想"我能实现什么"，你会产出一个实现细节丰富但接口错误的系统。这在软件里会让代码无法集成，在工作中让努力无法产生贡献。

**三类贡献对应软件团队的三类输出**：
- 直接成果 → 发布的功能
- 价值标准建立 → 代码库质量、工程文化、review标准
- 培养后继者 → mentoring、知识传递、技术债务的长期管理

三类缺任何一个，团队短期看起来正常，长期必然出问题。只做功能的团队，技术债务会拖垮它。不培养人的团队，知识集中在少数人，这是单点故障。

**对教学的直接应用**：

我每年教一遍相同的课程，算不算培养后继者？不算——如果教的内容不变，我只是在重复生产同样规格的学生。培养后继者意味着每次教学都是在改进"培养下一代能力"的接口，学生毕业后能继续独立成长。
