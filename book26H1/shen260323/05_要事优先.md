## 第二步：裁判循环

### 放弃昨天——用垃圾回收来理解

**老德这里说得绕，我翻译成自己的语言：**

"放弃昨天"本质上是一个显式垃圾回收（explicit garbage collection）问题。

在手动内存管理的语言（比如C）里，内存不会自动释放。过期的对象，如果你不手动free()，它们会一直占用内存，直到程序崩溃或内存耗尽。

"昨天的工作"就是这些死引用（dangling references）——曾经有用，现在没有价值了，但它们仍然占用你的时间、精力、注意力。它们不会自动消失，因为没有GC（garbage collector）。你需要手动释放。

**教学里的"昨天"问题**：

我的数据结构课程已经开了8年。每年我都会加新内容（最近几年加了图算法、加了并查集、加了更多动态规划题型），但几乎从没删过任何东西。

8年积累的结果：这门课的内容量是第一年的三倍，但学时没变。我每年都在以更快的速度讲越来越多的内容，学生的理解深度在下降。

原来这才是真正的问题：不是"我要加什么新内容"，是"我要删掉什么旧内容才能给新的重要内容腾空间"。

这就是"昨天"——并查集当年很重要，现在考试考吗？就业用到吗？如果答案是"很少"，它就是占用内存的死引用，需要free()。

**解决方案：零基础课程设计（类似zero-based budgeting）**

不是每年在上一版本上修改，而是每隔几年从空白开始：
- 这门课结束时，学生需要能做什么？（重新定义接口）
- 要达到这个接口，最少需要哪些内容？（从头设计实现）
- 逐项比对当前课程内容，不能justify的删掉

这个过程很痛苦，因为要放弃花了很多时间备课的内容。但不做这个，课程会越来越重，学生负担越来越高，理解越来越浅。

### 要事的判断

**正确的判断方式**：不是"这件事重要吗？"，是"如果现在不做，未来损失是否不可逆？"

紧急的事情不一定是要事。如果一个学生一直问同一类型的问题，每次我单独回答他（紧急处理）——但这类问题一直会有，真正的要事是改善课程里讲这个知识点的方式（解决根因）。

要事的清单不能超过3件。超过3件说明我还没做真正的选择，只是列了一个比较重要的清单。

## 第三步：结构

```
当前工作清单（包含所有"昨天"的累积）
    ↓
放弃审计：如果这项工作今天不存在，我会重新启动它吗？
    NO → 候选free()，找人接手或直接停止
    YES → 保留并排序
    ↓
排序后的保留清单（应该≤3件）
    ↓
专注执行：同一时间段只做一件（单线程）
    ↓
真实输出
```

## 第四步：可执行模型

```
IF 工作清单超过5项且感到分散
THEN 做放弃审计：问"如果今天不存在这项工作，我会重新启动它吗？"NO则free()

IF 每年课程内容只增不减
THEN 每隔2-3年做一次零基础课程重设计：先定义接口，再设计实现，现有内容逐项justify

IF 遇到"一切都很重要，没法选"
THEN 这是放弃恐惧伪装成判断困难。强迫自己问：如果只能做一件，不做哪件损失最不可逆？
```

## 第五步：接入CS知识体系

**同构：垃圾回收（GC）**

Java等语言有自动GC，但GC是有代价的——它会在某个时刻暂停程序（Stop-the-World），集中清理死对象。这个暂停是不可避免的，问题只是"何时暂停"和"付多少代价"。

如果你不做"放弃昨天"，你的工作系统等于没有GC的手动内存管理。死引用越积越多，系统越来越慢，最终OOM（Out Of Memory）——精力耗尽，什么都做不好。

定期做放弃审计，相当于主动触发GC，在受控条件下清理，而不是等到系统崩溃被迫清理。

**同构：单线程执行模型**

JavaScript是单线程的，它通过事件循环（event loop）处理并发，但任何一个时刻只有一个任务在CPU上运行。这不是缺陷，是设计选择——它消除了线程安全（thread safety）问题，让程序行为可预测。

复杂认知工作应该采用单线程模型：同一时间只推进一件重要工作。看起来像"低效"，实际上消除了context switch代价，总产出更高。

**接入教学**：

优先级不是排序问题，是删除问题。

学生经常问我"怎么安排学习优先级"，我之前的回答是"按重要性排序"。但这个答案是错的。排序假设所有事情都要做，只是顺序不同。真正的优先级决策是：哪些东西根本不应该出现在清单里？删掉之后，剩下的清单自然就能处理了。

这对课程设计同样成立：不是"如何把所有重要内容都塞进去"，是"删掉哪些内容才能让剩下的内容被真正学会"。
