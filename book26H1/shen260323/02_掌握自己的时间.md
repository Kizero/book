# 第2章：掌握自己的时间

## 第零步：ER图

```
时间块 {
    int  时长_分钟
    bool 物理不被打断
    bool 心理连续性    "上一件事的余震消散了"
}

时间浪费源 {
    enum 类型  "不必要会议 | 信息系统失灵 | 重复劳动 | 人员配置错误"
    string 实质  "全是组织问题伪装成个人问题"
}

有效工作 {
    string 类型  "复杂决策 | 创造性输出 | 深度分析"
    int    最小时间块_分钟  "通常≥90"
}

时间块   }|--|| 有效工作   : "是前提"
时间浪费源 }|--|| 时间块  : "侵蚀"
```

三步操作序列：记录 → 分析浪费源 → 整合为整块。顺序不能颠倒，因为你不知道自己在浪费什么，直到你记录了。

## 第二步：裁判循环

### 整块时间

**这是本章最关键的概念，但我最初理解错了。**

**第一直觉（错的）**：整块时间 = 日历上没有会议的时间段。

有一次我把上午10:00-12:00全部清空，准备写期末考题。结果：10:00坐下来，脑子里还在想9:50结束的课里一个学生的问题没回答清楚，花了20分钟处理这个余震。11:30开始想12:30有个组会，提前焦虑，实际专注时间约50分钟。

日历上是120分钟，实际是50分钟。

这就是为什么整块时间有三个条件：
1. 足够长（≥90分钟）
2. 物理不被打断
3. **心理不被污染**——上一件事结束了，下一件事焦虑还没开始

条件3是最难的，也是最容易被忽视的。

**OS类比**：认知工作是批处理（batch processing），不是分时处理（time-slicing）。

操作系统的分时调度适合I/O密集型任务——任务等I/O的时候CPU可以去做别的。但深度认知工作是CPU密集型：你在思考时，没有"等I/O"的间隙，切换就是浪费。而且认知工作的context switch cost极高——就像CPU缓存（cache），切换任务时缓存失效，下次回来要重新load所有上下文。

所以时间碎片化对认知工作的伤害，比表面看起来更严重。不是"少了一小时"，是"两个30分钟的产出，远低于一个60分钟的产出"，因为每次切换都付出了cache miss的代价。

**教学例子**：

我试过两种批改作业的方式：
- 方式A：在零散时间里批改，每次10-15分钟，批一两份就被打断
- 方式B：集中3小时，连续批完一批

方式B的批改质量完全不同。连续批改时，我的评分标准会在前5份作业里自动校准，之后保持一致；我能发现共同的错误模式，在最后统一写一个班级反馈。方式A批完的作业，评分标准飘移严重，我自己都发现同一个错误给了不同的分。

这不是"更努力"的问题，是批处理 vs 分时处理的架构问题。

### 时间浪费的四个来源

**核心结构**：德鲁克列的四个来源，全是组织问题伪装成个人问题。

| 来源 | 伪装形式 | 实质 |
|------|---------|------|
| 不必要的会议 | "我们需要对齐" | 授权或信任结构缺失 |
| 信息系统失灵 | "我需要问一下" | 信息流设计有缺陷 |
| 重复劳动 | "这个上次做过" | 职责边界不清 |
| 人员配置不当 | "我得替他做" | 招聘或培训问题 |

**关键判断**：如果一个时间浪费可以靠"个人更努力"消除，它不在这个列表里。这四个全需要组织层面的修复。

## 第三步：结构

```
现实：时间被切碎（默认状态）
    ↓
步骤1：记录（3-4周，不改变行为，只观察——建立基准数据）
    ↓
步骤2：分析（哪个来源造成了最多损失？组织问题还是个人问题？）
    ↓
步骤3：整合（把碎片合并成整块，满足三个条件）
    ↓
整块时间 → 有效工作成为可能
```

步骤1不能省。你以为的时间使用方式，和实际的，几乎肯定不同。必须用数据打破幻觉。

## 第四步：可执行模型

```
IF 从未做过时间记录
THEN 先记录1周（每小时一次），不改变任何行为，只建立基准数据

IF 每天深度工作时间 < 90分钟
THEN 找主要浪费源：会议？信息不畅？替别人填坑？针对来源修复，不是靠意志力撑

IF 需要做复杂工作（出题/设计课程/架构决策）
THEN 预留整块时间（≥90分钟），保证物理不被打断 + 前后留缓冲消除心理污染
```

## 第五步：接入CS知识体系

**同构：OS的batch processing vs time-slicing**

批处理系统（batch processing）把相同类型的任务攒在一起处理，减少context switch开销。认知工作应该采用批处理架构——相同类型的任务（批改作业、回复邮件、写代码）集中处理，不要混在一起。

time-slicing让每个任务看起来都在"进行"，实际上每个任务都在付出cache invalidation的代价。知识工作的"缓存"是工作记忆中的上下文，一旦切换，重建这个上下文的时间是真实成本，但它是隐形的——它不出现在日历上，但它消耗了时间和认知资源。

**接入教学**：

课程设计也有这个问题。如果一学期的课程内容是零散主题的拼凑（每节课都是新话题），学生每次上课都在付context switch cost，无法建立深层连接。好的课程设计是批处理架构：几节课围绕一个核心概念，让"缓存"有时间预热和巩固。

**张力**：

德鲁克的方法设计于1960年代。那时候"关上门不接电话"是可行的。现在组织的通信期望是即时响应，整块时间需要的不只是个人意志，还需要组织层面的规范支撑。德鲁克没有处理这个问题，这是他的时代局限。
