## 第二步：裁判循环

### 比较优势——德鲁克这里说得绕，我翻译成CS语言

**原始问题**：如果开发者A在前端和后端都比开发者B强，应该如何分配工作？

德鲁克的答案（来自比较优势原理）：让A做那个delta最大的。

用数字举例：
- A做前端：产出100，A做后端：产出80
- B做前端：产出60，B做后端：产出70

如果A做后端，B做前端：总产出 80+60=140
如果A做前端，B做后端：总产出 100+70=170

A虽然在两方面都比B强，但A的前端优势（100 vs 60，差40）大于后端优势（80 vs 70，差10）。让A做前端，B做后端，总产出更高。

这在经济学里叫比较优势（comparative advantage），但德鲁克讲的比李嘉图更直接：不问"谁更强"，问"delta最大在哪里"。

**教学例子**：

我们课程组有两个老师，我和另一个同事。我在算法设计和系统课都能教，他主要擅长系统课。按照"谁不忙谁上"的逻辑，我什么都能上，所以什么都来找我。

结果：算法课我讲得好，系统课讲完凑合，但没有一门课达到最好状态。

如果按比较优势分配：我专注算法课（我的delta最大的地方），同事专注系统课（他的唯一长处区），两门课整体质量都会更高。

这不是偏心，是资源优化。

### 职位设计

**核心原则**：先写接口（职位描述），再找实现（符合条件的人）。不能先有人再写接口。

**判断标准**：
```
测试：如果这个人明天离职，这个职位还需要存在吗？
→ YES：职位设计正确，围绕工作需求
→ NO：职位是为人量身定制的，需要重新设计
```

**教学例子**：

有一次我们需要设一个"算法竞赛辅导"职位。负责人说：李老师擅长竞赛，我们就让他来负责这个吧，然后再写职位描述。

这就把顺序搞错了。正确的顺序是：先定义"这个职位需要完成什么任务、需要什么能力"，再看谁符合。反过来做，职位变成了李老师能力的镜像——当李老师离职，这个职位的描述就没有意义了。

### 利用上司的长处

**这条最反直觉，但对我来说实际上就是API设计。**

上司是我需要调用的API。每个API都有它最适合处理的输入类型。如果我把错误格式的输入传给API，得不到好的输出，是我的使用方式有问题，不是API的问题。

上司是阅读型（偏好书面材料）还是听汇报型（偏好口头对话）？这决定了我应该以什么格式"传参"。

## 第三步：结构

```
用人决策流程：
1. 先问：这项工作需要什么能力（定义接口）
2. 再问：谁的delta最大（比较优势）
3. 配置：把工作放在那个人的长处区
4. 检查：短处是否会遮蔽长处？是→处理到够用；否→接受
5. 不断问：如果这个人离职，职位描述还有意义吗？
```

---

## 第四步：可执行模型

```
IF 要给人分配工作
THEN 不问"谁还没做过"，问"谁在这里的delta最大"

IF 下属有明显短处
THEN 先判断：短处是否遮蔽了长处？是→处理到够用即止；否→接受并绕过

IF 要设计或填充一个职位
THEN 先写清楚工作要求（接口），再找匹配的人（实现）——不能反过来

IF 需要从上司那里得到好的决策
THEN 先识别：他偏好书面还是口头？善于什么类型的判断？以最顺畅的方式"传参"
```

## 第五步：接入CS知识体系

**同构：接口优先设计（Interface-first design）**

好的软件设计先定义接口，再实现：
- 接口 = 职位要求（这个职位需要什么）
- 实现 = 具体的人（谁来做）
- 依赖倒置原则（DIP）：高层模块不应该依赖低层模块的实现，应该依赖接口

德鲁克的职位设计原则就是DIP的人力资源版本：组织不应该依赖特定的人，应该依赖职位接口。当实现（人）更换，接口（职位）保持稳定，组织继续运转。

**比较优势就是资源分配的优化问题**：

在CS里，这是一个assignment problem（分配问题），可以用匈牙利算法或线性规划求解。德鲁克的直觉解法和最优化的形式化方法指向同一个结论：最大化总产出，不是让每个人都做他最擅长的事，而是让每个人做他在整个团队中相对优势最大的事。

**接入教学**：

课程设计应该先于教师选择，不能反过来。如果先有教师再设计课程，课程内容会向教师的知识边界靠拢，而不是向学生需要学什么靠拢。这是典型的把实现凌驾于接口之上。
