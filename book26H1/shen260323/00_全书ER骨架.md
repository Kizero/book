# 第零步：全书ER骨架——《卓有成效的管理者》

## 这本书描述什么结构？

不是"如何管理别人"。是一个知识工作者如何在四个不可消除的约束内，让自己的工作产生真实的外部成果。输入是时间和脑力，约束是组织现实，输出在系统外部被感受到。

---

## 第一次ER草图（错的）

第一反应：把"有效性"当实体来建模。

```
有效性 {
    type: 可学习的实践
    result: 组织绩效提升
}
管理者 }|--|| 有效性 : "实践达到"
时间管理 ||--|{ 有效性 : "构成"
贡献 ||--|{ 有效性 : "构成"
```

画完就感觉不对。"有效性"的属性是什么？它和其他实体的独立关系是什么？全是空的。

这在数据库设计里是一个经典错误：把状态（state）reify成了实体（entity）。`user.is_active = true` 里的 `active` 不需要是一张独立的表，除非它有自己独有的属性和关系。"有效性"就是 `管理者.state`，由五个维度的执行决定。把它单独建成实体，只是增加了一个没有信息量的中间节点。

**修正：有效性是管理者的状态属性，不是独立实体。**

---

## 正确的ER图

```
管理者 {
    string 定义     "需要对组织绩效负责的知识工作者（不等于有下属）"
    bool   有效性   "状态属性，由五个维度的实践决定"
    list   四个约束 "时间受限 | 环境不可控 | 依赖他人 | 视野受限"
}

时间块 {
    int  时长_分钟
    bool 心理连续性  "不只是日历上的空白"
}

贡献 {
    enum 类型       "直接成果 | 价值标准建立 | 培养后继者"
    string 接收方   "组织外部或组织整体"
}

长处配置 {
    enum 主体  "自己 | 下属 | 上司"
    string 任务 "分配到该主体长处区的工作"
}

决策 {
    enum 问题性质  "经常性 | 例外"
    enum 当前状态  "意见阶段→检验阶段→推行阶段→反馈阶段"
}

管理者 ||--o{ 时间块      : "整合为"
管理者 ||--o{ 贡献        : "聚焦于"
管理者 ||--o{ 长处配置    : "执行"
管理者 ||--o{ 决策        : "做出"
时间块    }|--|| 决策     : "整块时间是复杂决策的前提"
贡献      }|--|| 决策     : "贡献方向决定决策的边界条件"
长处配置  }|--|| 贡献     : "通过长处放大贡献效能"
```

---

## 全书逻辑链

```
四个约束是常数（抱怨无意义，在内部设计行为）
    ↓
先掌握时间 → 没有时间块，其他四步全是幻觉
    ↓
方向：我能贡献什么 → 时间用在正确方向
    ↓
放大器：发挥长处 → 同样的时间，产出更多
    ↓
收缩：要事优先，删除昨天 → 时间集中在少数高价值事项
    ↓
质量保障：有效决策 → 高价值事项的输出质量可靠
    ↓
管理者.有效性 = true
```

五个维度不是并列清单，是依赖链。时间是基础资源，其他四个都依赖它。

## 完成标志（行为层）

- 五个维度中，哪个是其余四个的前置依赖？（如果答不上来，逻辑链没建立）
