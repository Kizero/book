# 第1章：卓有成效是可以学到的

## 第零步：ER图

```
知识工作者 {
    string 产出方式  "靠大脑，不靠体力"
}
    ||
    |--(子类)--> 管理者 {
                    string 定义  "需要对组织绩效负责的知识工作者"
                    bool   有效性  "状态属性"
                    list   约束   "时间受限 | 环境不可控 | 依赖他人 | 视野受限"
                }
```

管理者是知识工作者的子类，不是独立实体。有效性是状态，不是实体。就这两条。

## 第二步：裁判循环

### 有效性

**第一直觉（错的）**：一个开发者测试覆盖率100%，有效吗？

我当时判断：是，他把测试做到了极致。

**哪里错了**：100%覆盖率是"写测试这个过程"的efficiency。有效性问的是另一层：这些测试测的是正确的东西吗？测试通过了，软件在生产环境能正常工作吗？覆盖率是实现指标，有效性是目标指标。覆盖率100%而生产事故频发，这是高效率的无效性。

这就是Effectiveness vs Efficiency：
- Efficiency：把这件事做好了
- Effectiveness：这件事本身是对的事

类比到我自己的教学：我可以把一堂递归的课讲得极其流畅，板书漂亮，学生坐在那里猛点头。这是lecture efficiency。但如果课后让他们独立写一个递归程序，大多数人写不出来——这就是ineffective teaching。我把错的事情做好了。

**正例**：
- 课前先问"这节课结束后，学生能做到什么"，而不是"我要讲什么内容"
- 代码审查不问"这个PR写完了吗"，问"合并之后，系统能力有什么变化"

**边界例**：
- "我按时完成了所有里程碑" → 未必有效，如果项目方向本身错了
- "这学期我讲完了所有考纲内容" → 讲完 ≠ 学生会了

**边界定义**：有效性 = 在正确方向上产生可被外部感受到的成果。三个词缺一个都不成立。

---

### 管理者（德鲁克定义）

**正例**：
- 一个没有下属的助教，他的评分标准影响全班对"什么是好代码"的理解 → 是管理者
- 一个架构师，他的接口设计决定影响十几个下游团队的工作方式 → 是管理者

**边界例**：
- 课程组长，有行政权但所有技术决策都被上级否定，他的判断从未被执行 → 名义管理者，实质上不是

**反例伪装**：
- "我带着30人的团队" → 有下属不等于是管理者，关键是决策是否影响组织绩效

## 第三步：结构

```
四个约束（结构性常数，不能消除，只能在内部设计）
    ↓
选择A：被约束驱动 → 高效的无效性（忙碌但方向不对）
选择B：在约束内主动设计 → 有效性（可学，可实践）
```

四个约束的设计感：这不是随机列表。每一个都对应知识工作的一个结构性特征——时间不可再生、组织是共同依赖体、知识专业化导致视野局限。德鲁克在说：这就是你工作的物理定律，不是抱怨对象。

## 第四步：可执行模型

```
IF 感到忙碌但说不清楚贡献了什么
THEN 你在做效率，不是有效性。问：这件事完成后，系统外部会有什么不同？

IF 认为"我没有权力所以无法有效"
THEN 德鲁克定义的最大误读。有效性是个人实践，不依赖职级。

IF 觉得这本书的道理"我早就知道"
THEN 背诵不是理解，裁判新例子才是检验标准。
```

## 第五步：接入CS知识体系

**同构：correctness vs fitness for purpose**

软件工程里有两个不同的质量概念：
- Correctness：程序符合规格说明（spec）
- Fitness：规格说明本身是正确的（做了对的事）

一个程序可以完全符合一个错误的spec。Correctness ≠ Fitness。

德鲁克的Effectiveness = Fitness，Efficiency = Correctness。"没有什么比高效地做根本不该做的事更无用"——这句话翻译过来就是：一个correctly implemented的程序，如果spec写错了，那correctness毫无价值。

这个类比让我彻底理解了为什么有效性先于效率。你要先确认spec对不对，再谈实现质量。

**接入我自己的教学框架**：

我教学生建模的第一步是"搞清楚领域问题"，然后才建模。这就是effectiveness first——先确认问题是什么，再谈解法多好。如果第一步就错了，模型越精确越有害（它会让你对错误的问题越来越有信心）。

**张力**：

德鲁克说有效性"可以学到"，但对于学习路径语焉不详。他描述了系统，但没有描述学习曲线。这是这本书的真正局限——它告诉你what，不告诉你how to learn。
