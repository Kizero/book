# 第零章：全书ER骨架——《卓有成效的管理者》

这本书描述什么结构？一个知识工作者在四个不可消除的约束内，让自己的工作产生真实的外部成果。输入是时间和脑力，约束是组织现实，输出在系统外部被感受到。管理者不是职级，是角色定义。

---

## 第一次ER草图（错的）

第一反应：把"有效性"建成实体。

```mermaid
erDiagram
    EFFECTIVENESS {
        string type "可学习的实践"
        string result "组织绩效提升"
    }
    MANAGER {
        string name
        string role
    }
    TIME_BLOCK {
        int duration_min
    }
    CONTRIBUTION {
        string type
    }
    MANAGER }|--|| EFFECTIVENESS : "实践达到"
    TIME_BLOCK ||--|{ EFFECTIVENESS : "构成"
    CONTRIBUTION ||--|{ EFFECTIVENESS : "构成"
```

画完就知道错了。EFFECTIVENESS的属性是什么？它和其他实体的独立关系是什么？全是空的。这在ER设计里是经典的state reification错误——把状态（state）当成实体（entity）来建模。`user.is_active = true` 里的 `active` 不需要是一张独立的表，除非它有自己独有的属性和关联。"有效性"就是 `MANAGER.state`，由五个维度的执行决定。建成独立实体只是增加了一个没有信息量的中间节点，查询时还要多一次join，毫无收益。

**修正：有效性是管理者的状态属性，不是独立实体。**

---

## 正确的全书ER图

```mermaid
erDiagram
    MANAGER {
        string id
        string definition "需要对组织绩效负责的知识工作者，不等于有下属"
        bool effectiveness "状态属性，由五个维度的执行决定"
        string constraints "时间受限 / 环境不可控 / 依赖他人 / 视野受限"
    }
    TIME_BLOCK {
        string id
        int duration_min
        bool physically_uninterrupted
        bool psychologically_continuous "上一件事的余震消散了"
    }
    CONTRIBUTION {
        string id
        string type "direct_result / standard_setting / successor_development"
        string recipient "组织外部或组织整体"
    }
    STRENGTH_CONFIG {
        string id
        string subject "self / report / boss"
        string task "分配到该主体长处区的工作"
        float delta "此人在此任务上的相对优势值"
    }
    DECISION {
        string id
        string problem_nature "recurring / exception"
        string boundary_condition "必须满足X，不能损害Y"
        string state "opinion / boundary / option / implementation / feedback"
    }
    MANAGER ||--o{ TIME_BLOCK : "整合为"
    MANAGER ||--o{ CONTRIBUTION : "聚焦于"
    MANAGER ||--o{ STRENGTH_CONFIG : "配置"
    MANAGER ||--o{ DECISION : "做出"
    TIME_BLOCK }|--|| DECISION : "整块时间是复杂决策的前提条件"
    CONTRIBUTION }|--|| DECISION : "贡献方向决定决策的边界条件"
    STRENGTH_CONFIG }|--|| CONTRIBUTION : "通过长处放大贡献效能"
```

## 全书逻辑链（依赖链，不是并列）

```mermaid
graph TD
    A["四个约束是常数<br/>（抱怨无意义，在内部设计行为）"]
    B["掌握时间<br/>没有TIME_BLOCK，其余四步全是幻觉"]
    C["确定贡献方向<br/>时间用在正确接口上"]
    D["配置长处<br/>同样时间，delta最大化"]
    E["删除昨天<br/>显式GC，资源集中少数高价值项"]
    F["有效决策<br/>假说→证伪→推行→反馈"]
    G["MANAGER.effectiveness = true<br/>可持续，不是偶发"]

    A --> B
    B --> C
    C --> D
    D --> E
    E --> F
    F --> G
```

五个维度是依赖链。时间是基础资源，其他四个都以它为前提。没有整块时间，决策质量无从谈起。

## 完成标志（行为层）

两个问题：

一，给你一个知识工作者的工作场景，你能定位他的有效性瓶颈在哪个维度吗？

二，"有效性是实体还是属性"——你能给出精确的ER设计理由吗？不是"感觉不对"，是"这在ER设计里是state reification错误，因为它没有独立属性和独立关系"。

都答得出来，骨架完成。
