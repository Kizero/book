# 第零步：全书ER骨架——《卓有成效的管理者》

## 核心问题

这本书在描述什么？不是"管理他人的方法"。是一个知识工作者如何让自己的输出产生真实结果的系统。约束条件：组织现实。

---

## 第一次ER尝试（错的）

第一反应是把"有效性"建成一个实体：

```
管理者 → 实践 → 有效性
时间管理 → 支撑 → 有效性
贡献 → 构成 → 有效性
...
```

建着建着感觉不对。我在数据库设计里见过这种错误：把一个状态（state）reify成实体（entity）。"有效性"没有自己的属性，没有自己的关系，它只是管理者的一个状态值。就像数据库里不应该把 `is_active` 单独建表，除非它有自己的属性和关系。

**修正**：有效性是管理者的属性，不是独立实体。

---

## 正确的ER图

```
管理者 [effective: boolean, 四个约束: list]
    ||
    |--掌握--> 时间 [性质: 不可再生]
    |--聚焦--> 贡献 [三类: 成果/标准/人才]
    |--发挥--> 长处 [对象: 自己/下属/上司]
    |--设定--> 优先级 [原则: 删除>排序]
    |--做出--> 决策 [五要素状态机]
```

关键：这五个不是并列清单，是依赖链。没有整块时间，做不出好决策。没有贡献方向，优先级无从定义。

---

## 全书逻辑链

```
知识工作者面临四个结构性约束（不可消除）
    ↓
约束内的资源管理：时间整合
    ↓
资源有了，方向呢？问：我能贡献什么
    ↓
贡献需要发挥长处（自己+他人），不是平均分配能力
    ↓
长处有限，所以必须删除次要事项，集中
    ↓
集中精力做重大决策（有结构的决策，不是拍脑袋）
    ↓
以上持续实践 = 管理者的有效性
```

---

## 核心实体表

| 实体 | 关键属性 | 角色 |
|------|---------|------|
| 管理者 | 有效性（状态）、四个约束 | 主体 |
| 时间 | 不可再生、可整合 | 基础资源 |
| 贡献 | 三类成果 | 方向指针 |
| 长处 | 三类对象 | 放大器 |
| 优先级 | 删除机制、专注度 | 资源分配 |
| 决策 | 五要素、从意见开始 | 输出质量 |

注意：有效性不在表里。它是管理者的属性，不是独立行为体。这个区分在数据库设计里叫做"不要把派生属性建成实体"。
